## Build & Run

### iOS App
```bash
cd HealthyPlant
xcodegen generate                # generates .xcodeproj from project.yml
open HealthyPlant.xcodeproj      # build with Cmd+R
```
SPM dependencies (Lottie, Firebase) resolve automatically on first open. If not: File > Packages > Resolve Package Versions.

Requires `GoogleService-Info.plist` in `HealthyPlant/HealthyPlant/Resources/` (not committed).

**Install directly on device:**
```bash
# Get device ID (use xcrun xctrace, NOT xcrun devicectl — IDs differ)
xcrun xctrace list devices

xcodebuild \
  -project HealthyPlant/HealthyPlant.xcodeproj \
  -scheme HealthyPlant \
  -destination "id=<DEVICE_ID>" \
  -allowProvisioningUpdates \
  CODE_SIGN_STYLE=Automatic
```

**TestFlight upload (build number must be bumped in project.yml first):**
```bash
cd HealthyPlant
xcodegen generate
xcodebuild archive \
  -scheme HealthyPlant \
  -destination "generic/platform=iOS" \
  -archivePath /tmp/HealthyPlant.xcarchive \
  -allowProvisioningUpdates CODE_SIGN_STYLE=Automatic
xcodebuild -exportArchive \
  -archivePath /tmp/HealthyPlant.xcarchive \
  -exportPath /tmp/HealthyPlant_export \
  -exportOptionsPlist HealthyPlant/ExportOptions.plist
xcrun altool --upload-app \
  -f /tmp/HealthyPlant_export/HealthyPlant.ipa \
  --type ios \
  --apiKey <KEY_ID> --apiIssuer <ISSUER_ID>
```
App Store Connect API key credentials are stored in memory. `ExportOptions.plist` uses `destination: export` (not `upload`) so it works without an Xcode account session.

### Firmware (ESP32 Sensor)
```bash
cd firmware
pio run -e esp32           # build for NodeMCU ESP-32S
pio run -e esp32 -t upload # build + flash via USB
pio device monitor         # serial output at 115200 baud
```
- `platformio.ini` — `[platformio]` section sets `src_dir = healthyplant-sensor` (must be in `[platformio]`, not `[common]`)
- `config.h` — board selected via build_flags (`-DBOARD_ESP32` or `-DBOARD_ESP32S3`); includes the matching `config_espXX.h`
- `config_esp32.h` / `config_esp32s3.h` — pin mappings + API URL; guards are `CONFIG_ESP32_H` / `CONFIG_ESP32S3_H` (distinct, not `CONFIG_H`)
- NodeMCU ESP-32S wiring: Soil VCC→3V3, GND→GND, AOUT→P34 (GPIO34, ADC1)
- ESP32-S3 wiring: Soil VCC→3V3, GND→GND, AOUT→IO5 (ADC1)

**Sensor provisioning flow:**
1. Flash firmware → open serial monitor → note the **Sensor ID** printed on boot
2. In the app (Settings → Add Sensor): enter Sensor ID → register → receive **Device Token**
3. Power-cycle ESP32 → connect to the `HealthyPlant-XXXX` WiFi AP → captive portal opens → enter WiFi credentials + Device Token
4. Sensor begins posting readings to the API; link it to a plant profile in the sensor detail sheet

**Re-provisioning (change WiFi or token):** erase NVS first:
```bash
pio run -e esp32 -t erase && pio run -e esp32 -t upload
```

## Architecture

**Two-tier app**: SwiftUI iOS client + FastAPI Python backend on Cloud Run, with Firestore (NoSQL) and GCS for storage.

### iOS (MVVM, Swift 5.9)
- **App/** — `HealthyPlantApp` (Firebase init, AppDelegate for FCM/APNS), `ContentView` (tab bar + auth/onboarding gate), `AppState`, `AppEnvironment` (prod/dev URL switching via UserDefaults)
- **Models/** — `Codable` structs matching backend camelCase JSON
- **ViewModels/** — `@MainActor ObservableObject`, one per feature. Call services only, never `APIClient` directly.
- **Views/** — Feature folders: Home, Profiles, Garden, Community, Calendar, Assistant, Settings, Onboarding, Sensors, Scan
- **Services/** — `APIClient` (base HTTP + SSE), feature services, `AuthService` (Firebase anonymous + Google Sign-In), `ImageUploadService`, `NotificationService`
- **Utilities/** — `Theme` (dark theme, green accent `#00C853`), `ViewModifiers` (`.greenOutline()`), date/number extensions

Project generated by XcodeGen from `project.yml`. Bundle ID: `com.healthyplant.app`. Build number: `CURRENT_PROJECT_VERSION` in `project.yml`.

### Tab Structure
5 tabs (custom `TabBarView`): **Home** · **Profiles** · **Community** · **Calendar** · **Assistant**

- **Profiles tab** cycles three view modes: booklet → grid → garden (toggled by toolbar icon). Toolbar principal shows tappable rank badge → `RankLadderView` sheet.
- **Community tab** (`GardenView.swift` → `CommunityTabView`): segmented control switches between **Plants** (`CommunityBrowseView`) and **Gardeners** (`GardenersListView`). Tapping a gardener → `PublicGardenerProfileView`.
- **Settings** (gear icon on Home): sensor management — add, delete (swipe), view token (key button), view history chart + re-link plant (chart button → `SensorDetailView`)

### Backend (FastAPI, Python 3.12)
- **main.py** — CORS, auth middleware, router registration. Auth middleware checks `Authorization: Bearer <Firebase JWT>` first, falls back to `X-User-ID` header (anonymous users).
- **routers/** — `/api/v1/` prefix: `profiles`, `garden`, `calendar`, `chat`, `community`, `gardeners`, `sensors`, `auth`, `photos`
- **services/** — `FirestoreService` (all DB ops), `AIService` (Claude claude-sonnet-4-20250514 via Anthropic SDK), `StorageService` (GCS signed URLs), `FCMService` (push notifications via `firebase_admin`)
- **models/** — Pydantic v2 with camelCase `Field(alias=...)`. All response models use `model_config = {"populate_by_name": True}`.

### Auth & Identity
- Anonymous users: UUID generated on first launch, stored in `UserDefaults`, sent as `X-User-ID` header
- Google Sign-In: Firebase Auth JWT sent as `Authorization: Bearer` header; middleware sets `request.state.user_id` to Firebase UID
- **Account migration**: When anonymous user links Google, iOS calls `POST /api/v1/auth/migrate` with the old anonymous UUID. Backend batch-copies profiles, calendar events, chat messages, sensors, and the gardener profile root doc to the new Firebase UID.

### Firestore Schema
```
users/{userId}                        # gardener profile root doc (bio, experienceLevel, avatarURL, isPublic, fcmToken)
users/{userId}/profiles/              # plant profiles
users/{userId}/calendar_events/
users/{userId}/chat_messages/
users/{userId}/sensors/               # sensor docs (sensorId, deviceToken, linkedProfileId, ...)
users/{userId}/following/{targetId}   # follow graph
users/{userId}/followers/{followerId}
community_plants/{id}                 # denormalised shared plants (sourceUserId, gardenerAvatarURL, ...)
sensor_tokens/{token}                 # top-level lookup: {userId, sensorId} — avoids collection-group index
```

**Sensor token auth**: The ESP32 sends `X-Device-Token` on every reading POST. The middleware looks up `sensor_tokens/{token}` (O(1), no index needed) to resolve the owner's `userId` and `sensorId`. Avoid using a Firestore collection-group query for this — it requires a manually-created index and returns 401 if the index is missing.

### Key Data Flows
- **Profile creation**: POST returns immediately; AI recommendations (sun/water/harvest) generated async via `BackgroundTasks` and written back to Firestore
- **Chat**: SSE streaming — user message saved, history + all plant profiles loaded as context, Claude streams chunks, final message saved on `done` event
- **Calendar**: AI generates 7-day care schedule from all profiles; events deduplicated before insert
- **Gardener follow + notify**: Follow writes symmetric docs in both subcollections. On community plant share, `asyncio.create_task` fires FCM multicast to all followers' tokens via `send_each_for_multicast` (wrapped in `run_in_executor`).

### Gardening Rank System
Computed client-side in `GardeningRank.compute(profiles:)` from `sum(ageDays) + count × 30`. Tiers: Seedling (0) → Sprout (50) → Grower (200) → Gardener (500) → Green Thumb (1000) → Expert (2000) → Master Gardener (4000). Displayed in Profiles tab toolbar.

### Onboarding Gate (ContentView)
```
App launch
  └─ isSignedIn? no → SignInPromptView / anon continue
  └─ gardenerSetupComplete? (UserDefaults "hp_gardener_setup_complete") no → GardenerProfileSetupView
  └─ main TabBarView
```
FCM deep link (`"OpenCommunityTab"` NotificationCenter post) switches to the Community tab from anywhere.
